## PART 1 — Mega Evolution-era identity equivalence (production data bug: every me-series card gets zero PPT/PokeTrace comps)
Root cause verified upstream by the design layer 2026-07-04: PPT returns zero-padded collector numbers ("Tauros - 069/086"); PokeTrace returns `cardNumber: "096/086"` and `set.name: "ME04: Chaos Rising"` (code-prefixed). Catalog identity: `69/86` / `Chaos Rising`. Equivalence fails → identity gate rejects true matches. The Celebrations-bug protection is right; its rules are too literal.

- **F1 — Canonical collector-number equivalence.** ONE shared helper used everywhere numbers are compared: strip leading zeros from purely-numeric segments both sides ("069/086" ≡ "69/86"; "096/086" ≡ "96/86"), prefixed segments (TG06/GG30/SVP…) untouched. Apply in: PPT matching, PokeTrace matching, reconciler identity-gate comparisons, checked-comps matching, owned-sales matching, eBay Browse ask query builder (query printed form or both). Oracles: "069/086"≡"69/86"; "TG06/TG30" unchanged; "096/086" (secret rare above printedTotal) ≡ "96/86"; NO false equivalence "10/86" vs "100/86".
- **F2 — Provider set-name normalization.** Strip leading set-code prefixes matching `^[A-Z]{2,4}\d{1,3}:\s*` before comparison ("ME04: Chaos Rising" → "Chaos Rising"), case-insensitive compare after; add me-era sets to the alias table as needed. Negative oracle: a Celebrations-style same-number-different-set case must still hard-reject.
- **F3 — Pin the real PPT v2 payload shape** for these hits (number may be embedded in the name field with number/set fields absent) — parse whichever fields actually carry identity; commit the payload as a fixture. Don't guess.
- **F4 — Prove era-wide, end-to-end.** Add Tauros me4-69 RAW to the live-regression corpus. After deploy, probe production: must return priced PPT and/or PokeTrace candidates with a real headline (baseline 2026-07-04: headline null, all sources empty). Spot-probe one me-era card from a DIFFERENT me set. Report before/after JSON for both. Also sweep any script/doc still reading the comps response's old `results` key (renamed to `all`).
- **D1 (scanner binding):** the F1 helper must ALSO be used by the scanner's identity mapping shipped in 342f1e7 — me-era cards print zero-padded numbers. Oracle: scan identity `{name:"Tauros", number:"069/086"}` resolves to me4-69, same comp as typing "69/86". Same for F2 if scans yield provider-style set strings.

## PART 1b — Card art everywhere in comps (identity confirmation as a feature)
Ruling: every comp context shows the card's stock/catalog image wherever one can be found — the photo is the fastest "right card?" check a dealer has, independent of which source prices it.
- **Image resolution chain** (one shared helper): catalog imageUrl (scrydex/TCG API) → matched PokeTrace hit's `image` field → matched PPT image if exposed → styled art-missing placeholder (never a broken image or empty gap). Persist a resolved fallback URL onto the card's catalog record so it's found once, not per-lookup.
- **Display everywhere:** comp result header (verify it survives when catalog art is missing but a provider image exists), ambiguity alternatives (each candidate with its own art), deal-session line items, recent-comps list, buy watches, checked-comp logging sheet.
- **Rules:** image loading NEVER delays/blocks a comp; provider-CDN images are display-only and must NEVER enter eBay listing payloads or the photo pipeline (existing catalog-art listing rules unchanged); cache aggressively.
- **D2 (scanner binding):** the scan result screen shows the side-by-side — dealer's scan photo next to the resolved card's stock art, in the confirmed-match case as well as the ambiguity case. This is the half-second "right card?" confirmation that makes scanning trustworthy.
- **Tests:** no-catalog-image + PokeTrace-match renders the PokeTrace image; no image anywhere renders placeholder; listing-photo behaviour unchanged (negative test).

## PART 2 — Red-team the pricing brain (EXECUTABLE attacks; findings only — ZERO production-logic changes)
Build each attack as a test fixture, run it through the REAL reconciler/deal-calc code, report what actually happens. Deliverable: `docs/REDTEAM_2026-07.md` + committed attack-fixture suite, clearly named, EXCLUDED from main test gates.

Minimum attacks: (1) **Shill-bid poisoning** — thin eBay bucket, 3 of 5 "sales" the same inflated price: does the headline resist? (2) **Reprint crash** — true value halves overnight, all cached/stale sources show pre-crash prices: how long until honest, does anything warn? (3) **Checked-comp staleness trap** — 2 checked comps logged 80 days ago at a spike price, card since fell 40%: do they still headline at tier 0.90? (4) **Grade bleed** — PSA 9 prices leaking into a PSA 10 query via mislabeled source data. (5) **Currency shock** — GBP/USD moves 8% on a 6-day-old FX cache: quantify worst-case error on a £1,000 card. (6) **Deal-calc margin illusion** — sale prices near fee/postage-tier boundaries where rounding flips profit to loss. (7) **Owned-sales self-poisoning** — dealer's own 3 recent sales were a firesale 30% under market; tier-1.0 lowballs every future offer. (8) **At least 2 novel attacks you devise from reading reconciler.ts.**

Per attack: setup, expected honest behaviour, actual behaviour, verdict SURVIVES / DEGRADED / FAILS, raw output. Rank failures by money-at-risk for a dealer deploying ~£2,000. The FAILS/DEGRADED table goes to the design layer — implement NO fixes in this goal.
